# 要件ヒアリング記録

## 2026-07-20

### 確認済み

- アクセス制御情報はノートファイルから分離してよい。
- AsciiDocファイルから再構成できる必要があるのは、ノードの内容情報と
  ノード間の有向グラフである。
- ノートの保存場所は
  `<datadir>/<creator-user-uid>/<note-uid>.adoc` とする。
- ノートファイルのGit管理は当面行わない。
- 研究室全体を一つの空間として扱う。
- ノートごとに、閲覧、編集および管理の権限を設ける。
- 当初はユーザまたはグループへの権限付与を想定したが、後のヒアリングで
  初期バージョンはユーザへの直接付与だけに変更した。
- 新規ノートの作成者には管理権限を付与する。
- 新規ノートはデフォルトで非公開とし、作成者以外には権限を付与しない。
- 権限は `管理 > 編集 > 閲覧` の包含関係を持つ。
- 管理権限を持つユーザは、ノートの削除および権限変更ができる。
- 各ノートには一人以上の管理権限を持つユーザが必要であり、最後の管理権限は
  解除できない。
- 初期バージョンでは、ノートの権限をユーザへ直接付与する。
- グループおよびグループ単位のアクセス制御は初期スコープに含めない。
- 将来グループを導入する場合は、既存のユーザ権限と併存する機能として追加する。
- ノート単位の権限とは別に、ソフトウェア全体の管理者権限を設ける。
- 初期セットアップ時にシステム管理者である `root` ユーザを作成する。
- Web UI、API、OIDCコールバックおよびMCPを含め、サブパス配下での提供に対応する。
- `root` はOIDCとは独立した、パスワード認証のローカルアカウントとする。
- `root` はOIDC障害時にもログインできる緊急管理アカウントとする。
- `root` はユーザ管理、設定変更、インデックス再構築、全ノートおよびACLの
  管理を実行できる。
- `root` はノートACLを迂回できる。
- `root` による管理操作は監査ログへ記録する。
- `root` は削除、無効化および権限剥奪できない。
- 初期バージョンのノート削除はハードデリートとする。
- 削除前に、対象ノートを参照しているノート数を警告表示する。
- 削除時にAsciiDocファイル、データベース上の投影およびアクセス制御情報を削除する。
- 削除後に参照元ノートへ残った参照はリンク切れとして扱う。
- インデックス再構築時にリンク切れを検出し、警告として報告する。
- リンク切れでも、エッジが参照する元のUUIDは変更せず保持する。
- 参照先UUIDに対応するノートの有無を照合してリンク切れを判定する。
- リンク切れを特別なUUIDへ集約またはエイリアスしない。
- Nil UUIDは通常のノートIDとして使用できない予約値とする。
- ソフトデリートおよび復元は将来拡張として検討する。
- ノートIDと作成者ユーザIDは、ファイルパスだけでなくAsciiDoc内にも保持する。
- ACLおよび現在の管理者はAsciiDocに含めない。
- タグ機能は初期バージョンに含める。
- AsciiDocの解析およびHTML変換にはAdocWeaveを基盤として使用する。
- 本アプリのAsciiDoc機能プロファイルに適合させるためのAdocWeave拡張プログラムは、
  本アプリの実装範囲に含める。
- AsciiDocファイル全体のSHA-256ハッシュを、ノートのリビジョンとして使用する。
- リビジョンはAsciiDoc内へ保存しない。
- 更新および削除では、取得時のリビジョンを指定する。
- リビジョンが現在のファイルと一致する場合だけ変更を適用する。
- 競合時は変更を適用せず、現在のリビジョンを返す。
- 初期バージョンでは競合内容を自動マージしない。
- 更新日時は更新成功時にサーバが書き換える。
- ファイル更新は、一時ファイルへの書き込みとアトミックな置換によって行う。
- 有向エッジは `note:<target-note-uid>[<label>]` としてAsciiDoc本文中に記述する。
- マクロを記述したノートを始点、指定したノートを終点とする。
- 表示ラベルは任意とする。
- 初期バージョンでは関係種別を扱わず、有向エッジの有無だけを扱う。
- 自己ループを許可するが、グラフ描画時には表示しない。
- `note:<target-note-uid>#<anchor-id>[<label>]` により、参照先ノート内の
  AsciiDocアンカーを指定できる。
- ノート間参照には明示的なアンカーIDを推奨し、自動生成アンカーを使用する場合は
  見出し変更によるリンク切れの可能性を警告する。
- ノートが存在しないリンク切れと、ノートは存在するがアンカーが存在しない
  リンク切れを区別する。
- アンカーがリンク切れでも、ノート間の有向エッジは有効とする。
- 同じ始点と終点の参照はグラフ上で一本のエッジとして表示する。
- DBでは本文中の参照位置、参照先ノートIDおよびアンカーIDを個別に保持する。
- アンカーを独立したグラフノードとして描画しない。
- 初期バージョンではTypst記法による数式描画に対応しない。
- 初期バージョンでは、AsciiDoc標準のSTEM構文を使用したLaTeX記法の
  インライン数式とブロック数式に対応する。
- 初期バージョンでは、AsciiDoc標準のソースコードブロックと、言語に応じた
  シンタックスハイライトに対応する。
- ノートに記載されたソースコードは実行しない。
- Mermaid等による図表描画は将来拡張として検討する。
- 初期バージョンでは画像および一般ファイルのアップロードに対応しない。
- 添付機能の具体的な構文、保存場所およびデータモデルは、実装時に定義する。
- `http` および `https` の外部リンクを許可し、別タブで安全に開く。
- 危険なURLスキーム、`include`、生HTML、JavaScriptおよび外部画像の埋め込みを
  禁止する。
- サーバは外部リンク先の内容を代理取得しない。
- Tauriデスクトップアプリは初期スコープに含めない。
- 将来、Tauriを用いてLinux、WindowsおよびmacOS向けのデスクトップアプリを
  追加できるように設計する。
- デスクトップアプリは、Web版と同じバックエンドAPIおよび認証・認可モデルを
  再利用する。
- 各ユーザにはアプリケーション内部のUUIDを発行する。
- OIDCユーザの同一性は `(issuer, subject)` で判定し、メールアドレスや表示名を
  同一性判定に使わない。
- ファイル配置にはOIDCの `subject` ではなく、内部ユーザUUIDを使用する。
- OIDC上のユーザ情報が変わっても、内部UUIDとファイル配置は変更しない。
- `root` にもOIDCユーザとは区別された内部UUIDを割り当てる。
- OIDCユーザの初回登録は `open`、`approval`、`invite-only` から選択できる。
- 登録ポリシーはセットアップ時に選択し、`root` が後から変更できる。
- 非対話セットアップ時のデフォルトは `approval` とする。
- 招待はOIDC認証を省略せず、一回限り、有効期限付き、失効可能とする。
- 招待には対象メールアドレスを任意で指定できる。指定時はOIDCの検証済み
  メールアドレスとの一致を要求する。
- 招待トークンはハッシュ化して保存し、招待操作は監査ログへ記録する。
- `approval` でも有効な招待を使用したユーザは承認済みとして登録できる。
- SMTPを任意で設定し、招待メールを送信できる。
- SMTP未設定時は、招待リンクを手動で共有できる。
- Resend等を含むSMTP互換サービスを利用できる。
- SMTPパスワード等の秘密情報は環境変数またはsecretから取得し、DB、設定ファイル、
  画面およびログへ平文で保存・出力しない。
- 初期バージョンでは過去のAsciiDoc本文を保存せず、過去版の閲覧・復元機能を
  実装しない。
- ノート操作、ACL変更、ユーザ管理、招待、設定変更およびログイン結果を
  監査ログへ記録する。
- 監査ログには実行者、操作、対象ID、日時、成否およびリクエストIDを保存し、
  本文や秘密情報は保存しない。
- 監査ログを閲覧・エクスポートできるのは `root` だけとし、アプリ上からの編集や
  個別削除を認めない。
- 監査ログの保持期間は設定可能とし、デフォルトを365日とする。
- 初期バージョンでは本文履歴を保存しないが、将来、ノート形式およびAPIを
  破壊的に変更せず追加できる設計とする。
- ファイル操作を共通のサービス境界へ集約し、将来その境界で更新前の本文を
  履歴へ保存できるようにする。
- 将来の復元は、過去版の内容から新しい最新版を作成する操作とする。
- 履歴導入前の本文および削除済みノートは復元できない。
- 監査ログと本文履歴は独立した機能として扱う。
- 初期バージョンでは日本語と英語を主要な検索言語とする。
- 閲覧可能なノートについて、タイトル、本文、タグ、UUID、作成者、日時、
  参照先および被参照元から検索・絞り込みできる。
- タイトルの一致を本文より上位に評価する。
- 可読テキストに加え、コードブロックおよびLaTeX数式のソースも検索対象とする。
- Web APIとMCPで同じ検索能力を提供する。
- 初期バージョンでは一致箇所の抜粋を表示しない。
- ベクトル検索および意味検索は将来拡張とする。
- MCPはサブパス配下のStreamable HTTPで提供する。
- MCPではOAuth 2.1 Authorization Code FlowとPKCEを使用し、OIDCで認証した
  一般ユーザとして通常のノートACLを適用する。
- MCP用トークンのaudienceを検証し、`root` のMCP接続を許可しない。
- CUIおよびリモート環境向けにDevice Authorization Grantを優先拡張として設計する。
- Device Flowでは、別端末のブラウザでOIDC認証と承認を行えるようにする。
- 認証基盤の技術検証で実装負荷と相互運用性を評価し、Device Flowを初期リリースへ
  含めるか判断する。
- Device Flowを延期する場合は、初期版でヘッドレス認証を保証しない。
- Personal Access Tokenは初期スコープ外とし、必要性が判明した場合に追加する。
- 初期MCPでは `get_note`、`search_notes`、`create_note`、`update_note`、
  `delete_note` および `list_note_links` を提供する。
- 作成・更新ではタイトル、本文およびタグだけを入力として受け付け、ノートID、
  作成者および作成日時等はサーバが生成・維持する。
- 更新・削除には期待するリビジョンを要求し、削除には明示的な確認も要求する。
- 初期MCPではACL、ユーザ、招待、システム、監査ログ、`root` および一括操作を
  扱わない。
- ログイン後の主画面に、閲覧可能な全ノートの有向グラフを表示する。
- エッジの向きを無視した弱連結成分を計算し、最大成分を中央へ配置する。
- その他の連結成分と孤立ノードは周辺へ配置する。
- 最大連結成分の判定から、リンク切れと「権限なし」の仮ノードを除外する。
- 初期バージョンでは近傍グラフ専用画面を設けない。
- 検索結果は同じグラフ上で強調し、タグ等によって表示対象を絞り込める。
- ノードのクリックによってノートを選択・閲覧できる。
- 矢印、リンク切れ仮ノード、重複辺の集約、パン、ズーム、ドラッグに対応する。
- 自己ループは表示せず、レイアウトを初期バージョンでは永続化しない。
- 閲覧不能なノートのメタデータを表示しない。
- 閲覧可能なノートから閲覧不能なノートへの参照は匿名の仮ノードとして表示し、
  閲覧不能なノートからの参照は表示しない。
- `root` は全ノートを通常ノードとして表示できる。
- 初期運用規模は約15ユーザ、ユーザあたり約100ノート、合計約1,500ノートとする。
- 初期想定規模では、グラフ用メタデータとエッジを一括取得する。
- グラフ取得時に本文を含めず、ノート選択時に別途取得する。
- レイアウト計算によってUI操作を長時間停止させない。
- 大幅に規模が増えた場合の近傍表示、段階的読み込みおよび集約表示は将来拡張とする。
- 初期バージョンは単一アプリケーションインスタンスとSQLiteで動作する。
- SQLiteはWALモードで使用し、同一ホストのローカル永続ボリュームへ配置する。
- ノートの `datadir` とSQLiteファイルは別々に設定できる。
- Docker Composeを提供し、TLS終端は外部リバースプロキシへ委ねる。
- 公開URLとサブパスをBase URLとして設定する。
- 複数インスタンス、共有ストレージおよびPostgreSQLは将来拡張とする。
- 組み込みのバックアップアーカイブ作成機能は初期スコープに含めない。
- 初期バックアップは、アプリ停止中に `datadir` とSQLite DBを一緒にコピーする。
- secretはデータと別に運用者がバックアップする。
- バックアップのスケジュール、保存先および保持世代は外部で管理する。
- アーカイブ作成、オンラインスナップショット、世代管理および復元支援は
  将来拡張とする。
- バックアップ暗号化は低優先度の将来拡張とする。
- 初期バージョンで、全AsciiDocを検証して検索・グラフ用投影を再構築できる。
- 再構築ではユーザ、ACL、招待および監査ログを変更せず、検証成功後に
  トランザクションで投影を置換する。
- 初期バージョンで正式にサポートする編集経路はWeb UI、Web APIおよびMCPとする。
- `datadir` を通常ユーザへ公開しない。
- 初期バージョンでは `datadir` の外部変更をリアルタイム監視しない。
- 外部変更は `root` による明示的な投影再構築で反映する。
- 起動時にファイルと投影の不一致を警告するが、自動修復しない。
- 実ファイルからリビジョンを計算し、外部変更後の意図しない上書きを防止する。
- リアルタイム監視および自動再インデックスは将来拡張とする。
- 実行速度に関する具体的な数値目標は定めない。
- 単一インスタンスとし、無停止運用および可用性SLAを要求しない。
- 再起動および計画停止を許容する。
- 成功を返した変更は再起動後も維持し、異常終了によるファイルと投影の不一致を
  起動時に警告する。
- デフォルト上限は、ノート1 MiB、タイトル200文字、タグ64文字、ノートあたり
  50タグ、API・MCPリクエスト2 MiBとし、設定で変更可能とする。
- ブラウザによる初期セットアップ画面を提供しない。
- 未セットアップDBの初回起動時だけ `ROOT_PASSWORD` を必須とする。
- 初回起動時にパスワードをハッシュ化してDBへ保存し、平文を保存・出力しない。
- 初期化済みDBでは `ROOT_PASSWORD` を要求せず、指定されてもパスワードを
  自動変更しない。
- 初期バージョンでは `root` パスワードの復旧CLIを提供しない。
- 初期バージョンでは `root` パスワードの通常変更機能を提供しない。
- パスワードを紛失した場合は、バックアップからの復元またはDBの再初期化を必要とする。
- `root` パスワードの変更および復旧は将来拡張とする。
- `ROOT_PASSWORD_FILE` およびDocker secrets連携は将来拡張とする。
- Web UIの初期案は、中央のグラフ、左側の検索・タグ絞り込み、右側の
  ノート閲覧・編集パネルとする。
- Web UIは変動しやすい仮決定とし、プロトタイプとフィードバックに応じて変更する。
- UI変更がノート形式、Web APIおよびMCPへ波及しない設計とする。
- ノートIDおよび内部ユーザIDにはUUIDv7を使用する。
- 文書タイトルの直後にノートID、作成者ID、作成日時、更新日時およびタグを
  AsciiDoc属性として配置する。
- ノートID、作成者IDおよび作成日時は変更不可とし、更新日時はサーバだけが更新する。
- タグはカンマ区切りとし、空白を除去して、大文字・小文字を同一視する。
- 重複タグを統合し、正規化後の値によるロケール非依存の昇順へソートする。
- 初期バージョンでは `:stem: latexmath` を解釈するが、`stem` をノート識別用の
  固定メタデータとはせず、将来Typst等の独自数式マクロを併存可能にする。
- 作成日時および更新日時は、UTCのRFC 3339文字列、固定ミリ秒精度で保存する。
- UIでは利用者のローカルタイムゾーンへ変換して表示する。
- 有効化済みユーザはハードデリートせず、無効化および再有効化できる。
- 無効化時に全セッションとMCPトークンを失効し、ログインおよび操作を禁止する。
- 無効化しても作成者ID、監査ログおよびACLを維持し、再有効化時にACLを復帰させる。
- 唯一の有効な管理者となっているノートがある場合は、権限移譲まで無効化を拒否する。
- OIDCの `(issuer, subject)` を別ユーザへ再利用しない。
- 未承認の保留ユーザだけは完全削除できるが、関連する監査ログは維持する。
- 初期バージョンで設定できるOIDCプロバイダは一つとする。
- OIDCのissuer、client IDおよびclient secretを環境変数から設定し、secretを
  DBおよびログへ保存しない。
- Discoveryを利用し、リダイレクトURIをBase URLとサブパスから生成する。
- ID Tokenの署名、issuer、subject、audience、有効期限、発行日時、nonceおよび
  認可レスポンスのstateを検証する。
- 表示名は `name`、`preferred_username`、`email`、内部UUIDの順に選択する。
- 通常登録ではメールを必須とせず、メール指定招待でのみ検証済みメールを要求する。
- 複数OIDCプロバイダ対応は将来拡張とする。
- Web UIではサーバ側セッションを使用し、OIDCトークンをJavaScriptへ渡さない。
- セッションCookieに `Secure`、`HttpOnly`、`SameSite=Lax` およびサブパスに
  基づく `Path` を設定し、`Domain` 属性を設定しない。
- 状態変更リクエストでは、CSRFトークン、OriginおよびFetch Metadataを検証する。
- 一般ユーザのセッション期限は無操作24時間・絶対7日、`root` は無操作30分・
  絶対8時間をデフォルトとし、設定で変更可能にする。
- ログアウト時に即時失効し、`root` は全セッションの一覧表示・一括失効ができる。
- JSON REST APIをBase URL配下の `/api/v1/` で提供し、OpenAPI文書を用意する。
- 一覧・検索はカーソルページングに対応する。
- ノート取得で `ETag`、更新・削除で `If-Match` を使用する。
- 共通エラー形式と機械判定可能なコードを使用し、ACL違反は原則404とする。
- 後方互換な変更はv1内、破壊的変更は新しいメジャーバージョンで行う。
- Web UI、APIおよびMCPは同じRustのアプリケーションサービス層を利用する。
- 初期版では任意オリジンへのCORSを無効とする。
- Cookie認証とBearer認証を分離し、将来のTauriはOAuth tokenでAPIへ接続する。
- AsciiDocとSQLiteをまたぐ変更では、SQLiteに処理中操作を記録する。
- ファイル変更前の異常終了は操作を取り消し、ファイル変更後の異常終了は
  再起動時にSQLite側の更新を完了する。
- ファイルとDBの両方が完了するまでAPIは成功を返さない。
- 自動復旧を監査ログへ記録し、復旧不能時は該当ノートの変更を停止して警告する。
- アプリ自身の操作だけを自動復旧し、外部ファイル変更は明示的な投影再構築を要求する。
- ハードデリートは削除準備と削除確定の二段階で実行する。
- 削除準備では、閲覧可能な参照元の件数と、5分間・一回限り有効な確認トークンを返す。
- `root` にだけ全参照元の件数を表示する。
- 確認トークンを対象、実行者、リビジョンおよび被参照状態へ結び付け、
  DBにはハッシュだけを保存する。
- 準備後に状態が変化した場合は削除せず、再確認を要求する。
- Web UI、APIおよびMCPは同じ二段階削除サービスを使用する。
- 本アプリがMCP向けOAuth Authorization ServerおよびResource Serverを兼ねる。
- 外部OIDCは本人認証だけに使用し、外部tokenをMCPや他サービスへ転送しない。
- 本アプリが不透明なAccess TokenとローテーションするRefresh Tokenを発行し、
  DBにはハッシュだけを保存する。
- Access Tokenの期限は1時間、Refresh Tokenの絶対期限は30日をデフォルトとする。
- `notes:read`、`notes:write` および `notes:delete` scopeを提供し、
  scopeとノートACLの両方を検証する。
- OAuth Client ID Metadata Documentsと、`root` によるクライアント事前登録に
  対応する。
- CUI・デスクトップクライアントをpublic clientとして扱い、PKCE S256を必須とする。
- リダイレクトURIを完全一致で検証し、HTTPSまたはloopback localhostだけを許可する。
- 一般ユーザは自身の認可を確認・取消でき、`root` は全認可を強制失効できる。
- Dynamic Client Registrationは将来拡張とする。
