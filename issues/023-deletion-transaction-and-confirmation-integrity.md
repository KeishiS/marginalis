# 023: 削除トランザクションと確認トークンの整合性

状態: 実装済み。統合復旧試験は継続課題。

## 問題

`delete_confirmations.note_id`は`notes`を参照する外部キーである一方、確認トークンの消費は行を
削除せず`consumed_at_ms`を更新するだけである。このため、確認済みトークンがあるノートを物理削除すると、
正本削除後の`DELETE FROM notes`が外部キー制約で失敗する。正本だけが消え、投影とACLが残るため、
起動時復旧も同じ失敗を繰り返す。

さらにWeb APIの`DELETE /api/v1/notes/{note_id}`は、確認トークンを経由せず単段で削除する。
削除準備も被参照数を返さず、確認トークンへ被参照状態を固定していない。

## 実装項目

1. 削除確定時に、対象ノートの未使用・使用済みを問わない確認トークンを、投影削除と同じSQLite
   transactionで安全に消去する。または外部キーと履歴の設計を改め、物理削除を妨げない形にする。
2. Web APIを`prepare_delete_note`と`delete_note(confirmation_token)`の同一アプリケーションサービスへ
   移行し、単段DELETEを撤去する。OpenAPIも同時に更新する。
3. 削除準備でACLを考慮した被参照数を返し、対象ノートのrevisionと被参照状態のスナップショットを
   確認トークンに結び付ける。いずれかが変化した確定要求は競合として拒否する。
4. 正本、投影、ACL、xref、確認トークンおよびoperation journalを含む実SQLite・実filesystemの
   結合試験を追加する。prepare→confirm、失敗後のrecover、単段経路不在を検証する。

## 完了条件

- 確認済みトークンが存在しても、物理削除後に正本、投影、ACLおよび確認情報が一貫して除去される。
- 削除準備後の本文更新、削除、参照追加・削除のいずれも確認を無効化する。
- REST、MCPおよび将来のWeb UIが同じ二段階削除サービスだけを使用する。
