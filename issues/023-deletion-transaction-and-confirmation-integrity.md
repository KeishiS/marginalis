# 023: 削除処理と確認トークンの整合性

状態: 実装済み。結合復旧試験は継続課題。

## 目的

ノートの物理削除を必ず二段階の確認処理で実行する。正本、投影、ACL、参照、
確認トークン、操作記録を、一貫した状態で削除できるようにする。

## 解決する問題

`delete_confirmations.note_id`は、`notes`を参照する外部キーである。一方、
確認トークンを消費した際は、行を削除せず`consumed_at_ms`だけを更新していた。

この状態でノートを物理削除すると、正本を削除した後の`DELETE FROM notes`が
外部キー制約により失敗する。正本だけが削除され、投影とACLが残る。
起動時の復旧処理でも、同じ失敗が繰り返される。

さらに、次の問題があった。

- `DELETE /api/v1/notes/{note_id}`は、確認トークンを使わず一段階で削除していた。
- 削除準備は、対象ノートを参照している項目数を返していなかった。
- 確認トークンには、削除準備時の被参照状態が固定されていなかった。

## 実装方針

1. 削除確定時に、対象ノートの確認トークンを投影と同じSQLiteトランザクションで削除する。
   トークンが未使用か使用済みかは問わない。
2. Web APIを`prepare_delete_note`と`delete_note(confirmation_token)`からなる
   アプリケーションサービスへ移行する。
3. 確認なしの一段階削除を廃止し、OpenAPI文書も同時に更新する。
4. 削除準備では、ACLを考慮した被参照数を返す。
5. 対象ノートのリビジョンと被参照状態のスナップショットを、確認トークンへ結び付ける。
6. 削除準備後にリビジョンまたは被参照状態が変わった場合、削除確定を競合として拒否する。
7. 実際のSQLiteとファイルシステムを使う結合試験を追加する。

結合試験では、正本、投影、ACL、相互参照、確認トークン、操作記録を対象とする。
削除準備から確定までの成功経路、失敗後の復旧、一段階削除経路が存在しないことを確認する。

## 完了条件

- 使用済みの確認トークンが存在しても、物理削除後に正本、投影、ACL、確認情報が残らない。
- 削除準備後に本文を更新すると、確認が無効になる。
- 削除準備後にノートを削除すると、確認が無効になる。
- 削除準備後に参照を追加または削除すると、確認が無効になる。
- REST API、MCP、将来のWeb UIが、同じ二段階削除サービスだけを使用する。

実装は完了している。失敗後の状態を含む結合復旧試験は、継続課題として扱う。
