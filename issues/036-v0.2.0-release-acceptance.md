# 036: v0.2.0のリリース受入

## 状態

準備中。正式版の版更新、`v0.2.0-rc.1`以降の修正、追加の自動試験をリリース用ブランチへ
まとめている。自動判定、差分の実環境受入、Pull Request、タグ公開は未完了である。

## 目的

`v0.2.0-rc.1`で完了した新しいデータフォーマットv1の受入結果を基準に、正式版までの差分を
検証して`v0.2.0`を公開する。公開後はリリース候補版を作らず、`v0.x.y`形式の通常版を
順次公開する。

## 公開範囲

- アプリケーション版を`0.2.0`へ更新する。
- AdocWeaveのRust版とWASM版をv0.6.1の同一コミットへ固定する。
- `/api/v1`のOpenAPI仕様、MCPツール仕様、データフォーマットは
  `v0.2.0-rc.1`から変更しない。
- データフォーマットv1はAdocWeave v0.6.1前提である。以前のv1は移行せず、
  初回更新前に旧`dataDir`を削除して空の状態から初期化する。

## RC以降の差分

- HTTPミドルウェアの適用順を修正し、レスポンスの`X-Request-Id`とtracingスパンの
  request IDを一致させる。
- 利用者自身がMCP認可を取り消した後、既存のaccess tokenとrefresh tokenがHTTP境界で
  拒否されることを結合試験で固定する。
- CSRFの実装、E2E試験の実行条件、自動検証と手動受入の分担を文書へ反映する。
- Codexがflakeで固定した開発環境から検証を実行できるよう、
  `nix develop --command`を許可コマンドへ追加する。

## 自動判定

リリース用コミットで、Linux環境から次を実行する。

```bash
nix develop --command cargo make release-gate
```

この検査には、Rustの書式・型検査・Clippy・全テスト、依存境界、OpenAPI、
AdocWeaveの版と供給経路、GitHub Actionsの構文、flake評価、Nixパッケージ、
NixOSモジュールVM、実バイナリVMを含める。

## 差分の実環境受入

`v0.2.0-rc.1`の受入記録は[Issue 035](035-v0.2.0-rc.1-release-acceptance.md)に残す。
正式版候補では、[実環境での受入確認](../docs/acceptance.md)のうち次を再確認する。

1. `GET /api/v1/health`の`X-Request-Id`がUUIDv7であり、同じリクエストを記録した
   `journalctl -u marginalis.service`のrequest IDと一致する。
2. `/acceptance`からノートを一つ作成して削除できる。
3. MCP認可を取り消した後、既存のaccess tokenによる`/mcp`とrefresh tokenによる
   `/oauth/token`での再発行が拒否される。
4. 実行中のバイナリが`0.2.0`を報告し、`GET /api/v1/openapi.json`が公開仕様と一致する。

資格情報、Cookie、OIDCコード、トークン、ノート本文は記録しない。成否、実行日時、
候補コミット、必要な`X-Request-Id`だけを記録する。

## リリースを妨げる問題

- 自動判定または差分の実環境受入が失敗する。
- パッケージ群、flake、実行中バイナリ、`CHANGELOG.md`の版が一致しない。
- `X-Request-Id`とtracingスパンのrequest IDを対応付けられない。
- 認可取消後もaccess tokenまたはrefresh tokenを利用できる。
- RCで受け入れたOpenAPI、MCP、データフォーマットの互換性が変わる。

## 完了条件

- `Cargo.toml`群、`Cargo.lock`、flake、`CHANGELOG.md`が`0.2.0`で一致する。
- `cargo make release-gate`と差分の実環境受入が成功する。
- リリース用Pull Requestの必須チェックが成功し、`main`へマージされる。
- `CHANGELOG.md`の`未公開`を公開日へ置き換える。
- マージ済みのコミットへ注釈付きタグ`v0.2.0`を作成してpushし、タグで起動した
  GitHub Actionsの`Release gate`が成功する。
- GitHub Releaseを作成する場合は、タグの検証後に行う。
