# 019: Webセキュリティーの基準とroot管理境界

状態: 完了。

## 目的

Cookie認証を使用するREST APIとroot管理機能について、最低限のセキュリティー要件を
明確にする。リバースプロキシーを使用する構成でも、要件を検証できる状態にする。

## 解決する問題

- CSRF対策がトークン照合だけであり、`Origin`とFetch Metadataを検証していなかった。
- ログアウト時に削除するCookieの`Path`が`/`に固定されていた。
- 公開URLにサブパスがある場合、Cookieの発行範囲と削除範囲が一致しなかった。
- rootログインの回数制限がプロセス内かつTCP接続元単位だった。
- リバースプロキシー経由では、全利用者が同じ接続元として扱われる可能性があった。
- root用経路と通常経路が、同じルーターとリスナーに置かれていた。

## 採用した方針

1. Cookieの発行と削除に、同じパス方針を適用する。
2. 状態を変更する要求では、CSRFトークン、許可済み`Origin`、Fetch Metadataを検証する。
3. 転送ヘッダーは既定で信頼しない。
4. 利用者ごとの接続回数制限は、リバースプロキシー側で実施する。
5. Marginalisは、TCP接続元に対する補助的な回数制限だけを維持する。
6. root用経路を独立したルーターへ分離する。
7. 管理専用オリジンとmTLSへの対応は、後続の課題とする。

## 実施内容

- セッションCookieとCSRF Cookieを、発行時と同じ公開URLのサブパスで削除するようにした。
- サブパスを使用したCookieの発行と削除を試験で確認した。
- Cookieセッションを伴う変更操作では、CSRFトークン、固定した公開オリジン、
  `Sec-Fetch-Site`をすべて検証するようにした。
- クライアントIPにはTCP接続元だけを使用し、転送ヘッダーを信頼しない設計にした。
- リバースプロキシー配下の利用者別制限を、プロキシー側の責務としてNixOS運用文書へ記載した。
- `administration_router`には、rootログインと`/api/v1/admin/*`だけを収容した。
- 構成ルートで、管理ルーターを通常ルーターへ統合するようにした。

この構成により、将来は業務ロジックを移動せず、ルーターの配置変更だけで管理機能を
別リスナー、別オリジン、mTLSへ移行できる。

## 完了条件

- サブパス構成で、ログイン時とログアウト時のCookieが同じ`Path`を使用する。
- サイトをまたぐ状態変更要求が、CSRF、`Origin`、Fetch Metadataの要件を満たさなければ拒否される。
- リバースプロキシーの有無による回数制限の違いが、文書と試験で明確になっている。
- root用ルーターを別リスナーへ移す際に、HTTP経路の設定変更だけで対応できる。

上記の条件を満たしたため、このIssueを完了とする。
